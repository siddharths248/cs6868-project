Background:
Multiple threads are trying to access (and possibly modify) some shared object.
We need a consistent order in the end.
Can be achieved if all our threads agree on one order of execution.

What we do:
Each thread proposes to go in first, and our final choice is the "consensus" (since all other threads are agreeing on this).
The Universal Construction depends in this consensus. We maintain a global log of the proposed operations.

More stuff:

Consensus Object: provides a "decide()" method to all the threads, which in turn call it with their input v at most once.

When each thread calls the decide method, it returns a consistent (same across all the other threads) and valid (this is somethread's input) value.

We want a wait-free implementation of a consensus object. (Since the (a finite number of) threads can only call the method once, a lock-free 
implementation of the object
 is wait-free as well)

This means we want to see if there is a class of objects that is powerful enough to solve the consensus problem. (Examples of classes: CAS, TAS, 
read/write etc)

Now:
We don't really care about the number of objects (the hardware can handle it), we are only concerned with the properties of the class when it 
comes to
solveing the consensus problem.

==================================================================================
Definition 5.1.1. A class C solves n-thread consensus if there exists a consensus
protocol for n threads using any number of objects of class C and any number of
atomic registers.
-----------------------------------------------------------------------------------
Definition 5.1.2. The consensus number of a class C is the largest n for which that
class solves n-thread consensus. If no largest n exists, we say the consensus number
of the class is infinite.
-----------------------------------------------------------------------------------
Corollary 5.1.3. Suppose one can implement an object of class C from one or more
objects of class D, together with some number of atomic registers. If class C solves
n-consensus, then so does class D.
-----------------------------------------------------------------------------------
Both the below lemmas are existential lemmas
Lemma 5.1.4. Every two-thread consensus protocol has a bivalent initial state. 
(So basically the leaf values can either be 0 or 1 depending on the execution)
-----------------------------------------------------------------------------------
Lemma 5.1.5: Same for n-thread consensus (it has a bivalent initial state, that is if
input values are either zero or one).
==================================================================================

Critical state: The state where the valency changes in the next move

Every wait-free consenus protocol must terminate, all consensus protocols have a bivalent initial state -> every wait-free c.p has a crit. state.

ATOMIC REGISTERS 
================

Can't solve consensus problem.

==================================================================================
Theorem 5.2.1. Atomic registers have consensus number 1.
==================================================================================


Proof:
Consider we are AT the critical state (we were bivalent till now, and from the next state we enter univalent states)

Note that in all these cases, we are provinding an example of a case where the atomic registers fail to solve the consensus problem.

Case 1:
A reads from a register, while B reads/writes to the same/a different register.
Say A goes first (so input value is 0), then B enters (with its own input value) {Since A moved first, this state should be 0-valent}
The alternate case could be B going in first (and A not executing at all, say it crashed or something). {Since B moved first, this state should be 1-valent}
Both are different pathways from the initial state.

Note that in B's perspective, both the cases above are identical since A is only reading and no change will be made to the shared memory.
So, in both cases B should decide on the same value.
This leads to a contradiction since decide function should return a consistent value (0/1 depending on the pathway here, but B is giving the same one)


Case 2:
Suppose both the threads write to different registers (A to r0 and B to r1)
Now consider the case where A writes first and then B (This is the 0-valent pathway)
The second case would be B going first (the 1-valent pathway)
Now, after both have written, both the pathways must lead to the same state bcz, there is no way in which A or B could know who went first (bcz 
they're writing to different registers)
So again this is a contradiction since the next state is no longer univalent (since it was reached by both the pathways)

Case 3:
Both write to the same register.
Like the first case, say A went first and then B wrote to the register. In the other other scenario, say A crashed and never wrote anything, so B was
the first one to write. In either of the scenarios B is agnostic to what A did: first case it overwrites A's progress and in the second, A wasn't there,
So again, we face the same contradiction as in the first case.

==================================================================================
Corollary 5.2.2. It is impossible to construct a wait-free implementation of any ob-
ject with consensus number greater than 1 using atomic registers.
==================================================================================
